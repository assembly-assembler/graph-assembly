# bill-gist — 법안 카드 요약 부트스트랩

graph-assembly 사이트 카드의 법안 설명에 들어갈 **"개선 전 → 개선 후" 요약(gist)** 을 만든다.
원천은 의안정보시스템에서 적재한 **제안이유·주요내용 원문**(`snapshots/bill_summaries_22.json`).
gist는 `snapshots/bill_summaries_llm_22.json`에 `{ billId: { before, after } }` 형태로 누적 저장된다.
카드는 위에 **개선 전**(현행/배경), 가로선, 아래에 **개선 후**(이 법안이 바꾸는 것)를 보여준다.

## 절대 원칙 (CLAUDE.md 설계 원칙)
- **출처 필수·추측 금지**: gist는 **주어진 원문에 있는 내용만으로** 작성한다. 원문에 없는 사실·배경·평가를 절대 지어내지 말 것. 원문의 압축 요약일 뿐이다.
- **런타임 LLM 0**: 이 가공은 빌드타임 부트스트랩. 결과(JSON)는 커밋되어 사이트는 정적으로 조회만 한다.
- **한국어**, 사이트 톤(담백한 editorial). 정당/진영 평가·수식 금지.

## gist 작성 규칙
각 법안마다 `before`(개선 전)와 `after`(개선 후) **두 항목**을 만든다.
- **`before` = 현행/배경**: 지금 상태나 문제(원문의 "현행법은…", "…지적이 있음" 부분). 이 법안이 **바꾸기 전**의 모습.
- **`after` = 개선 후**: 이 법안이 **바꾸는 것**(원문의 "이에 … 하려는 것임" 부분). 동작 어미(`~하도록 함`, `~ 신설`, `~ 연장`, `~ 의무화`)로.
- 각각 **한 문장, 약 30~60자.** 카드에 한눈에 들어와야 함.
- 법안 제목을 그대로 반복하지 말 것(카드에 제목은 이미 있음). 조문 번호(예: "안 제40조의2")는 생략, 요지만.
- `before`와 `after`가 **대비**되게 쓴다(무엇이 어떻게 달라지는지). 원문에 배경 설명이 약하면 `before`는 현행 제도/공백을 간결히.
- 원문이 비었거나 "정보를 찾을 수 없습니다"류면 그 billId는 **건너뛴다**(출력에서 제외).

### 예시
- 원문: "…사업주는 인공지능 산업재해 예방시스템을 구축·운영할 수 있도록 하고, 정부는 …기술지원, 보조금의 지급 등 필요한 지원을 할 수 있도록 하려는 것임."
  → `{ "before": "산업재해 예방에 인공지능을 활용할 제도적 근거가 없음.", "after": "사업주의 AI 산업재해 예방시스템 운영과 정부의 기술·보조금 지원 근거를 신설." }`
- 원문: "…전기자동차 취득세 감면 특례가 2026년 말 종료될 예정이나 …보급 확대를 위해 …연장하려는 것임."
  → `{ "before": "전기차 취득세 감면 특례가 2026년 말 종료될 예정.", "after": "감면 특례를 2029년 말까지 3년 연장." }`

## 실행 절차 (한 배치)
1. 진행률 확인:
   ```
   npm run gist stats
   ```
2. 다음 배치(gist 없는 법안 원문)를 받는다. 한 번에 30~50건 권장:
   ```
   npm run gist next 40
   ```
   출력은 `[{ billId, name, raw }, ...]` JSON. `raw`가 제안이유·주요내용 원문.
3. 각 항목의 `raw`를 읽고 위 규칙대로 gist를 만들어, `{ "<billId>": { "before": "...", "after": "..." }, ... }`
   형태 JSON을 임시 파일에 저장한다. 예: `/tmp/gist-batch.json`. (원문이 부실하거나 before/after 중 하나라도 못 쓰면 그 billId는 넣지 않는다 — merge가 둘 다 있어야 받는다.)
4. 병합:
   ```
   npm run gist merge /tmp/gist-batch.json
   ```
5. 루틴이면 2~4를 반복(매 실행 1~수 배치). `npm run gist stats`의 "남음"이 0이면 완료.

## 주의
- `bill_summaries_22.json`이 아직 적재 중이거나 비었으면 먼저 `npm run ingest:summaries`로 원문을 채운다.
- gist는 누적·증분이다. 이미 있는 billId는 `next`가 다시 주지 않으므로, 신규 법안만 자연히 처리된다.
- 한 번에 너무 많이(>80건) 만들면 품질이 흐려진다. 배치를 작게 유지.
