# dev-loop — 지속 개발 루프 (1 사이클)

`BACKLOG.md`에서 **ready 작업 1개**를 골라 끝까지(검증·커밋·백로그 갱신) 처리한다.
시작 전에 `CLAUDE.md`의 **핵심 함정**과 **설계 원칙**을 숙지한 상태여야 한다.

## 사이클 절차

1. **선행 점검**: `git status`.
   - 변경이 문서·루프 세팅 파일(BACKLOG.md, `.claude/` 등)뿐이면 별도 커밋(`chore:`)으로 정리하고 진행.
   - 코드·데이터 변경 잔재가 있으면 새 작업을 시작하지 말고 상황만 보고하고 종료.
2. **작업 선택**: `BACKLOG.md`의 최상위 `ready` 1개. 한 사이클에 못 끝낼 규모면 백로그에서 작은 조각으로 쪼개 적어 두고 **첫 조각만** 실행.
3. **구현**: 데이터원·필드를 쓰는 작업은 **코드 작성 전에 라이브 검증**(curl로 실제 응답 확인 — `ingest/assembly/client.ts`의 curl 우회 패턴). 추측 엔드포인트/필드 금지.
4. **검증**: `npm run build` 통과는 **모든 사이클 필수**. 추가로 그 작업의 백로그 "완료 기준"을 충족. UI에 닿는 변경이면 `npm run preview`로 페이지 확인(dev 서버는 FOUC 때문에 쓰지 않음).
5. **백로그 갱신**: 완료 항목을 `done(YYYY-MM-DD) 결과 한 줄`로 바꾸고 done 섹션으로 이동. 작업 중 발견한 새 작업·버그·결정 필요 사항은 적절한 우선순위/상태로 추가.
6. **커밋**: 변경분(백로그 포함)을 **로컬 커밋만**. 메시지는 기존 스타일(`feat:`/`fix:`/`ci:`/`chore:` + 한국어 요약). **푸시 금지** — push = Pages 배포 트리거라 사용자가 직접 한다.
7. **보고**: ① 이번 사이클에서 한 일과 검증 결과 ② 다음 ready 작업 ③ 사용자 결정을 기다리는 `needs-user` 항목 — 을 한 단락으로.

## 가드레일 (위반 금지)

- **UI/디자인**: 사용자와 시안(목업) 합의 없이 보이는 화면을 바꾸지 않는다. `needs-user` 작업은 ASCII 목업·조사·제안까지만 만들어 백로그 노트에 남기고, 구현은 사용자 승인 후 `ready`로 바뀐 뒤에.
- **푸시·배포 금지**: `git push`, `gh pr`/release, workflow dispatch 금지. 로컬 커밋까지가 루프의 권한.
- **데이터 무결성**: 출처 필수·추측 금지. `snapshots/`는 적재 스크립트로만 갱신(손 편집 금지). 런타임 LLM 0 유지(LLM은 빌드타임 부트스트랩만).
- **대용량 diff**: snapshots 재적재로 수십 MB diff가 생기면 커밋하지 말고 보고만(커밋 여부는 사용자 판단).
- **의존성 추가**는 꼭 필요할 때만, 사유를 백로그 노트에 기록.

## /loop 페이싱 (자율 반복 시)

- `/loop /dev-loop`(동적 모드): 사이클 완료 후 다음 `ready`가 남았으면 **짧은 wakeup(60s)** 으로 바로 다음 사이클. 단 **연속 3사이클**을 마쳤으면 1200s 쉬어 사용자가 끼어들 틈을 둔다.
- **종료 조건** (wakeup을 잡지 않고 끝낸다):
  - `ready`가 0개 → "백로그 소진 — 루프 종료" + `needs-user` 대기 목록 보고.
  - 같은 원인으로 빌드/검증이 **2사이클 연속 실패** → 원인 보고 후 종료(사용자 판단 필요).
